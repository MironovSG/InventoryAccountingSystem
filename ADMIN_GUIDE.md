# Руководство администратора - Система учета ТМЦ

## Обзор

Администратор системы имеет **полный доступ** ко всем функциям и данным системы. Это включает в себя управление пользователями, мониторинг системы, просмотр логов аудита, управление базой данных и все функции МОЛ.

---

## Вход в систему

1. Откройте браузер и перейдите по адресу: `http://localhost:8080/api/` или: `http://localhost:8080/api/index.html`
2. Введите логин и пароль администратора
3. Нажмите кнопку **"Войти"**
4. Вы будете автоматически перенаправлены на **панель администратора** (`/admin-dashboard.html`)

**Тестовый администратор:**
- **Логин:** `admin`
- **Пароль:** `admin123`

## Главная панель администратора

### Панель мониторинга (`/admin-dashboard.html`)

После входа вы увидите главный экран с ключевыми параметрами системы:

#### 1. Состояние системы

**Статус системы:**
- **Работает** (зеленый) - система функционирует нормально
- **Ошибки** (красный) - обнаружены проблемы
- **Предупреждения** (желтый) - есть незначительные проблемы

**Информация об uptime:**
- Показывает время работы системы без перезагрузки
- Автоматически обновляется

**Нагрузка на систему:**
- Процент использования процессора
- Использование памяти (RAM)
- Активные подключения к базе данных

#### 2. Статус базы данных

**Индикаторы:**
- **Подключение** - статус подключения к PostgreSQL
- **Размер базы** - общий размер базы данных
- **Количество таблиц** - число таблиц в схеме
- **Последний бэкап** - дата последнего резервного копирования

**Действия:**
- Кнопка **"Обновить"** - обновить данные о состоянии системы

#### 3. Статистика системы

**Пользователи:**
- Всего зарегистрированных пользователей
- Активных пользователей
- Неактивных пользователей
- Пользователей по ролям (ADMIN, MOL, MANAGER, ENGINEER)

**Материалы:**
- Всего материалов в каталоге
- Материалов с низким запасом (требуют пополнения)
- Материалов отсутствующих на складе
- Количество категорий материалов

**Заявки:**
- Всего заявок в системе
- Заявок в ожидании обработки
- Обработанных заявок
- Отклоненных заявок

**Последние заявки:**
- Таблица с последними 10-20 заявками
- Информация о номере, дате, статусе, заявителе

## Управление базой данных

### Просмотр структуры БД (`/admin-database.html`)

Страница позволяет просматривать все таблицы базы данных и их содержимое.

#### Доступные таблицы:

1. **users** - Пользователи системы
   - ID, логин, пароль (хешированный), email, роль, подразделение, рабочая группа, статус активности

2. **departments** - Подразделения
   - ID, код, название, описание, родительское подразделение

3. **materials** - Материалы/ТМЦ
   - Код, название, категория, склад, расположение, остаток, минимальный запас, единица измерения, цена

4. **material_categories** - Категории материалов
   - ID, код, название, описание, родительская категория

5. **requests** - Заявки на выдачу ТМЦ
   - Номер заявки, дата, статус, заявитель, подразделение, назначение, примечания

6. **request_items** - Позиции в заявках
   - Связь с заявкой, материал, запрошенное количество, выданное количество

7. **material_movements** - Движения материалов
   - Тип операции (поступление, выдача, списание, корректировка), дата, материал, количество

8. **audit_logs** - Журнал аудита
   - Пользователь, действие, тип объекта, дата, описание, IP-адрес

9. **notifications** - Уведомления
   - Получатель, тип, статус прочтения, дата создания

#### Просмотр данных таблицы:

1. Выберите таблицу из списка
2. Нажмите **"Просмотреть"**
3. Откроется таблица с данными:
   - Все записи с пагинацией
   - Возможность прокрутки
   - Отображение всех столбцов

**Важно:**
- Просмотр данных доступен **только для чтения**
- Изменение данных напрямую в БД не рекомендуется (используйте интерфейс приложения)
- Для прямого редактирования БД используйте инструменты PostgreSQL (pgAdmin, DBeaver)

## Журнал аудита (Логи)

### Просмотр журнала (`/admin-logs.html`)

Журнал аудита содержит **все действия пользователей** и **системные события**.

#### Структура записи лога:

| Поле | Описание |
|------|----------|
| **ID** | Уникальный идентификатор записи |
| **Дата и время** | Когда произошло действие |
| **Пользователь** | Кто выполнил действие (логин) |
| **Действие** | Тип действия (CREATE, UPDATE, DELETE, LOGIN и т.д.) |
| **Тип объекта** | На какой сущности выполнено действие (User, Material, Request) |
| **ID объекта** | Идентификатор затронутого объекта |
| **Описание** | Детальное описание действия |
| **Старое значение** | Значение до изменения (для UPDATE) |
| **Новое значение** | Значение после изменения (для UPDATE) |
| **IP-адрес** | С какого IP выполнено действие |

#### Фильтры журнала

**1. Фильтр по датам:**
- **Дата с:** - начало периода
- **Дата по:** - конец периода
- Нажмите **"Применить фильтр"** для фильтрации

**2. Фильтр по типу события:**
Выпадающий список с типами действий:
- **Все события** - показать все
- **CREATE** - создание записей
- **UPDATE** - обновление записей
- **DELETE** - удаление записей
- **LOGIN** - входы в систему
- **LOGOUT** - выходы из системы
- **APPROVE** - одобрение заявок
- **REJECT** - отклонение заявок
- И другие типы действий

**3. Комбинированные фильтры:**
- Можно использовать фильтры одновременно
- Например: показать все UPDATE за последнюю неделю

#### Экспорт логов

**Два формата экспорта:**

1. **Экспорт в PDF:**
   - Нажмите кнопку **"Экспорт в PDF"**
   - Выгрузится файл с логами в формате PDF
   - Содержит все записи с учетом примененных фильтров
   - Форматированная таблица с заголовками

2. **Экспорт в Excel:**
   - Нажмите кнопку **"Экспорт в Excel"**
   - Выгрузится файл `.xlsx` с логами
   - Удобен для анализа в Microsoft Excel
   - Сохраняет все столбцы и данные

**Примечание:**
- Экспортируются только записи, видимые в текущем виде (с учетом фильтров)
- Если фильтры не применены - экспортируются все записи

#### Автоматическое обновление

- Таблица логов автоматически обновляется каждые 30 секунд
- Новые записи появляются вверху таблицы
- Можно обновить вручную кнопкой **"Обновить"**

## Резервное копирование базы данных

### Рекомендации по бэкапам

**Автоматическое резервное копирование:**

1. Настройте регулярное копирование PostgreSQL:
```bash
# Пример cron задачи для ежедневного бэкапа (Linux)
0 2 * * * pg_dump -U postgres inventory_db > /backup/inventory_$(date +\%Y\%m\%d).sql
```

**Ручное резервное копирование:**

1. Используйте `pg_dump`:
```bash
pg_dump -U postgres -d inventory_db -F c -f backup.dump
```

2. Или через pgAdmin:
   - Правый клик на базе данных
   - Выбрать "Backup"
   - Указать путь для сохранения

**Восстановление из бэкапа:**

```bash
pg_restore -U postgres -d inventory_db -c backup.dump
```

## Безопасность

### Лучшие практики

1. **Пароли администратора:**
   - Используйте сложные пароли (минимум 12 символов)
   - Регулярно меняйте пароль администратора
   - Не используйте один пароль для нескольких систем

2. **Контроль доступа:**
   - Назначайте роль ADMIN только доверенным лицам
   - Минимум администраторов (рекомендуется 2-3)
   - Используйте учетные записи с индивидуальными логинами (не общие)

3. **Мониторинг действий:**
   - Регулярно просматривайте журнал аудита
   - Обращайте внимание на подозрительную активность
   - Проверяйте изменения ролей пользователей

4. **Резервное копирование:**
   - Выполняйте регулярные бэкапы БД
   - Храните бэкапы в безопасном месте
   - Тестируйте восстановление из бэкапа

5. **Обновления системы:**
   - Следите за обновлениями Spring Boot и зависимостей
   - Тестируйте обновления на тестовом окружении
   - Применяйте обновления безопасности немедленно

## Устранение проблем

### Проблема: Система не запускается

**Диагностика:**
1. Проверьте логи приложения:
   - Файл: `logs/inventory-system.log`
   - Или консоль при запуске

2. Проверьте подключение к БД:
   - Убедитесь что PostgreSQL запущен
   - Проверьте параметры подключения в `application.yml`

3. Проверьте порт:
   - По умолчанию приложение использует порт 8080
   - Убедитесь что порт свободен

**Решение:**
```bash
# Проверка подключения к БД
psql -U postgres -d inventory_db

# Проверка доступности порта (Windows)
netstat -ano | findstr :8080
```

### Проблема: Пользователь не может войти

**Диагностика:**
1. Проверьте статус пользователя в таблице `users`
2. Проверьте логин и пароль
3. Проверьте роль пользователя

**Решение:**
1. Откройте "Управление пользователями"
2. Найдите пользователя
3. Проверьте что "Активен" = ✓
4. Если нужно - сбросьте пароль через редактирование

### Проблема: База данных не отвечает

**Диагностика:**
1. Проверьте статус PostgreSQL:
```bash
# Windows (если установлен как служба)
sc query postgresql-x64-13
```

2. Проверьте логи PostgreSQL

**Решение:**
1. Перезапустите службу PostgreSQL
2. Проверьте место на диске (БД не сможет работать при нехватке места)
3. Проверьте логи PostgreSQL на ошибки

### Проблема: Медленная работа системы

**Диагностика:**
1. Проверьте использование ресурсов (CPU, RAM) на панели мониторинга
2. Проверьте количество записей в таблицах
3. Проверьте наличие индексов в БД

**Решение:**
1. Увеличьте память JVM (параметр `-Xmx`):
```bash
java -Xmx2048m -jar inventory-system-1.0.0.jar
```

2. Оптимизируйте базу данных:
   - Создайте индексы на часто используемых столбцах
   - Очистите старые записи из `audit_logs` (если их много)

3. Проверьте сетевую задержку при подключении к БД

## Конфигурация системы

### Файл `application.yml`

Основные настройки системы находятся в `src/main/resources/application.yml`:

#### База данных:
```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/inventory_db
    username: postgres
    password: your_password
```

#### JWT настройки:
```yaml
jwt:
  secret: your-secret-key-change-in-production
  expiration: 86400000  # 24 часа в миллисекундах
```

#### Порог низкого запаса:
```yaml
inventory:
  low-stock-threshold: 10  # Минимальное количество для предупреждения
```

**Важно:** Не коммитьте файл `application.yml` с реальными паролями в Git. Используйте переменные окружения или профили Spring.

## Миграции базы данных

### Flyway миграции

Система использует Flyway для управления схемой БД.

**Файлы миграций:** `src/main/resources/db/migration/`

**Существующие миграции:**
- `V1__initial_schema.sql` - начальная схема БД
- `V2__insert_initial_data.sql` - тестовые данные
- `V3__add_new_features.sql` - дополнительные функции

**Создание новой миграции:**

1. Создайте файл: `V4__description.sql`
2. Напишите SQL скрипт
3. При следующем запуске приложения миграция применится автоматически

**Важно:** 
- Имена файлов должны быть уникальными и идти по возрастанию (V1, V2, V3...)
- Не изменяйте уже примененные миграции
